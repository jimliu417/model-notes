# OpenClaw QMD 降本增效笔记

## 1. 这篇文章主要讲什么？

这篇文章主要讲的是：

> OpenClaw 通过 QMD 记忆系统，把原本很长的上下文变成少量相关片段，从而减少 token 消耗、提升响应速度、降低 API 成本。

简单理解：

以前 AI 助手为了记住历史内容，可能会把一大堆聊天记录、记忆文件、项目资料都塞进上下文。

但问题是：

```text
上下文越长
→ token 越多
→ 响应越慢
→ API 成本越高
→ 越容易超时或触发 rate limit
```

QMD 的作用就是：

```text
不要把所有记忆都塞给 AI
而是先在本地搜索最相关的几句话
只把真正相关的内容交给 AI
```

---

## 2. 先理解问题：为什么 OpenClaw 会变慢？

OpenClaw 这类 AI 助手通常需要“记忆”。

比如你之前和它聊过项目、配置、需求、代码，它要回答新问题时，就需要参考历史内容。

传统做法可能是：

```text
把 MEMORY.md 或聊天历史全部塞进上下文
```

这会带来几个问题：

1. 很多内容和当前问题无关；
2. 上下文越来越长；
3. 每次请求都要处理大量 token；
4. 响应速度变慢；
5. API 成本变高；
6. 长会话容易卡死或超时。

比如你只是问：

```text
我们之前那个项目最后用了什么方案？
```

但系统可能把几个月的聊天记录全塞进去。

这就像你问别人一个小问题，结果让他先翻完一本几百页的笔记本。

---

## 3. QMD 是什么？

QMD 可以理解成 OpenClaw 的一种“本地记忆检索系统”。

它的核心作用是：

```text
从大量记忆中，快速找出和当前问题最相关的内容。
```

也就是说，QMD 不直接替代大模型，而是在大模型回答之前，先帮它筛选记忆。

整体流程大概是：

```text
用户提问
  ↓
QMD 在本地记忆中搜索相关内容
  ↓
只取最相关的片段
  ↓
把这些片段和用户问题一起发给 AI
  ↓
AI 根据更少、更准的上下文回答
```

所以 QMD 解决的是：

```text
上下文太长、无关信息太多、token 消耗太高的问题。
```

---

## 4. QMD 和传统记忆方式的区别

### 传统方式

```text
用户提问
  ↓
把完整 MEMORY.md / 聊天历史塞给 AI
  ↓
AI 在大量内容中自己找重点
  ↓
token 多，响应慢，成本高
```

### QMD 方式

```text
用户提问
  ↓
QMD 先本地搜索
  ↓
找到最相关的几句话
  ↓
只把相关片段发给 AI
  ↓
token 少，响应快，成本低
```

区别可以总结成一句话：

> 传统方式是“全量塞进去”，QMD 是“先检索，再精准提供”。

---

## 5. 为什么 QMD 能降低 token？

因为 AI 不再需要读取所有历史内容。

假设原来有：

```text
10000 token 的历史记忆
```

但当前问题真正相关的可能只有：

```text
200 token
```

如果不使用 QMD，AI 每次都要处理 10000 token。

使用 QMD 后，系统只把那 200 token 的相关片段发给 AI。

所以 token 会明显下降。

这就是文章里说的：

```text
上下文 token 可以削减 60% - 97%
平均可能超过 90%
```

---

## 6. 为什么 token 少了，速度就会变快？

大模型处理请求时，输入 token 越多，耗时通常越长。

可以简单理解为：

```text
AI 要读的内容越多
→ 处理时间越长
→ 费用越高
→ 出错概率越高
```

所以如果从几千、几万 token 降到几百 token，响应速度自然会快很多。

文章里的例子大概可以理解为：

| 场景 | 原来 | 使用 QMD 后 |
|---|---|---|
| 日常对话 | 需要处理很多历史上下文 | 只处理相关片段 |
| 长期会话 | 容易超时或卡死 | 可以正常响应 |
| 跨文件查询 | 上下文很大 | 先检索相关内容 |
| API 成本 | 随 token 增加而上涨 | token 少，成本下降 |

---

## 7. QMD 的技术原理

文章里提到，QMD 使用的是“混合搜索”思路。

可以理解成三层：

### 1. 关键词搜索

类似传统搜索。

如果你问：

```text
之前用的数据库是什么？
```

系统会优先找包含“数据库”“方案”“项目”等关键词的内容。

优点是：

```text
精准、速度快
```

缺点是：

```text
如果表达方式不同，可能搜不到
```

---

### 2. 语义搜索

语义搜索不是只看关键词，而是看意思是否接近。

比如你问：

```text
之前那个系统最后选了什么技术？
```

它也可能找到：

```text
我们最终采用了 Spring Boot + MySQL + Redis 的方案。
```

即使里面没有完全一样的关键词，也能通过语义相似度找到。

优点是：

```text
能理解意思相近的表达
```

缺点是：

```text
有时可能不够精确
```

---

### 3. 重排序

搜索得到多个候选结果后，再重新判断哪个最相关。

也就是：

```text
先粗略找出一批可能相关内容
再把最相关的排在前面
```

这一步可以提升最终返回内容的准确性。

---

## 8. QMD 的整体理解

QMD 不是简单的关键词搜索，也不是单纯的向量搜索。

它更像是：

```text
关键词搜索 + 语义搜索 + 重排序
```

组合起来的本地记忆检索系统。

它的目标不是生成答案，而是帮助 AI 找到更准的上下文。

## 9. 在 OpenClaw 里怎么启用 QMD？

### 前提条件

OpenClaw 版本需要满足要求。

先检查版本：

```bash
openclaw --version
```

如果版本太低，需要先更新 OpenClaw。

---

### 安装 QMD

文章里给出的安装方式是：

```bash
npm i -g bun
bun install -g github:tobi/qmd
```

安装后检查：

```bash
qmd --version
```

如果能显示版本号，说明 QMD CLI 安装成功。

---

### 安装 SQLite

QMD 需要 SQLite 支持。

不同系统安装方式不同。

#### Ubuntu / Debian

```bash
sudo apt update
sudo apt install sqlite3 libsqlite3-dev
```

验证：

```bash
sqlite3 --version
```

#### Windows

Windows 可以用 Chocolatey：

```powershell
choco install sqlite
```

也可以手动下载 SQLite 工具包，然后把路径添加到系统环境变量 PATH。

---

### 修改 OpenClaw 配置

找到 OpenClaw 配置文件。

Windows 路径一般类似：

```text
C:\Users\你的用户名\.openclaw\openclaw.json
```

macOS / Linux 路径一般类似：

```text
~/.openclaw/openclaw.json
```

在配置里加入：

```json
{
  "memory": {
    "backend": "qmd",
    "qmd": {
      "limits": {
        "timeoutMs": 8000
      }
    }
  }
}
```

这段配置的意思是：

```text
memory.backend = qmd
表示 OpenClaw 使用 QMD 作为记忆后端。

timeoutMs = 8000
表示 QMD 检索最多等待 8 秒。
```

---

### 重启 OpenClaw

配置完成后重启：

```bash
openclaw gateway restart
```

或者在聊天里发送：

```text
/restart
```

然后查看日志：

```bash
openclaw logs --follow
```

如果日志里出现类似：

```text
Using QMD memory backend
```

就说明 QMD 已经启用。

---

## 10. 启用 QMD 后有什么变化？

可以从四个方面理解。

### 1. 速度更快

因为上下文变短了，大模型要处理的内容少了。

### 2. 成本更低

API 通常按 token 计费。

token 越少，成本越低。

### 3. 稳定性更好

长上下文容易触发：

```text
rate limit
context overflow
timeout
卡死
```

减少上下文后，这些问题会明显减少。

### 4. 回答可能更准

因为无关信息少了，AI 更容易专注于真正相关的内容。

这点很重要：

```text
上下文不是越多越好
相关上下文才有价值
```

如果给 AI 一堆无关内容，反而可能干扰回答。

---

## 11. 什么情况下特别适合启用 QMD？

如果你有以下情况，就很适合使用 QMD：

1. OpenClaw 长期运行；
2. 有大量历史聊天记录；
3. 经常需要查询之前的项目资料；
4. 经常遇到响应慢；
5. 经常触发 rate limit；
6. API 成本越来越高；
7. 机器人部署在飞书、钉钉、企业微信等长期使用场景；
8. 希望 AI 能记住历史，但又不想每次塞全部内容。

尤其是长期运行的 Agent，如果不做记忆检索，很容易出现上下文膨胀问题。

---

## 12. QMD 会不会影响回答质量？

从文章的观点看，QMD 不但不会降低质量，反而可能提升质量。

原因是：

```text
它过滤掉大量无关内容
只保留和当前问题有关的上下文
```

AI 拿到的信息更干净，回答自然更容易聚焦。

但是也要注意：

```text
如果 QMD 没检索到关键内容，AI 仍然可能回答不完整。
```

所以 QMD 的效果取决于：

1. 记忆是否完整；
2. 检索是否准确；
3. 问题表达是否清楚；
4. 配置是否正常。

---

## 13. QMD 需要联网吗？

按文章描述：

```text
首次下载模型需要联网。
模型下载完成后，可以本地离线运行。
```

这也是 QMD 的一个优点：

```text
记忆检索过程在本地完成
不会额外消耗 API token
数据也不会因为检索而上传到外部服务
```

---

## 14. QMD 占用多少空间？

文章中提到 QMD 会下载本地模型。

可以简单理解为：

```text
QMD 需要占用一定磁盘空间
主要是 embedding 模型、重排序模型和查询扩展模型
```

所以它不是完全“零成本”。

它节省的是：

```text
API token 成本
响应等待成本
长上下文带来的稳定性成本
```

但它会占用：

```text
本地磁盘空间
本地计算资源
首次模型下载时间
```

---

## 15. 如果 QMD 出问题怎么办？

OpenClaw 会回退到默认记忆系统。

你也可以手动把配置改回 SQLite：

```json
{
  "memory": {
    "backend": "sqlite"
  }
}
```

然后重启 OpenClaw。

这说明 QMD 更像一个可选增强组件，而不是唯一依赖。

## 16. 这篇文章给我的核心启发

这篇文章不只是讲 OpenClaw 的配置，更重要的是讲了一个通用 AI 工程问题：

> 不要把所有历史内容都塞给模型，应该先检索相关内容，再把少量高质量上下文交给模型。

这其实就是很多 AI Agent、RAG 系统、长期记忆系统都会遇到的问题。

核心思想可以总结成：

```text
全量上下文不可持续
相关上下文才有价值
```

## 17. 我自己的理解

过去我可能会觉得：

```text
AI 上下文越多越聪明
```

但这篇文章提醒我：

```text
上下文不是越多越好
而是越相关越好
```

如果把大量无关信息都塞给 AI，会造成三个问题：

1. 慢；
2. 贵；
3. 容易被干扰。

更好的方式是：

```text
先检索，再注入上下文。
```

这和 RAG 的思想很像：

```text
用户问题
  ↓
检索相关资料
  ↓
把相关资料交给模型
  ↓
模型基于资料回答
```

QMD 本质上就是把这个思路用在 OpenClaw 的记忆系统里。

---

## 18. 这篇文章可以抽象出的通用原则

### 原则 1：不要全量塞上下文

能检索就先检索。

### 原则 2：上下文要少而准

宁可给 3 条强相关内容，也不要给 300 条无关内容。

### 原则 3：长期记忆必须可检索

如果记忆只是一个越来越大的文件，迟早会拖垮系统。

### 原则 4：本地检索可以省 token

本地检索不消耗大模型 API token，适合长期运行的 Agent。

### 原则 5：速度、成本、准确率是一起优化的

减少无关上下文，不只是省钱，也能提升稳定性和回答质量。

---

## 19. 和 RAG 的关系

QMD 可以看成一种面向 OpenClaw 记忆系统的轻量 RAG。

普通 RAG 是：

```text
用户问题 → 检索知识库 → 模型回答
```

QMD 是：

```text
用户问题 → 检索本地记忆 → 模型回答
```

二者核心思想一致：

```text
不要让模型凭空记忆所有内容
而是按需检索相关信息
```

---

## 20. 一句话总结

QMD 的核心价值是：

> 用本地检索替代全量上下文，把“什么都塞给 AI”变成“只给 AI 最相关的信息”。

这样可以减少 token、提升速度、降低成本，并减少长会话带来的卡死和超时问题。
